#### 😀代码逻辑与目的：
该代码片段用于初始化一个AI代码评审服务的请求对象。其核心逻辑是构建一个包含角色设定的消息列表，发送给大模型。本次修改将提示词的角色从 `user`（用户）修正为 `system`（系统），这在对话模型中用于设定AI的行为模式和人设，确保模型优先遵循该指令进行代码评审，是符合Prompt Engineering最佳实践的修正。

#### ✅代码优点：
1. **角色设定正确**：将人设和指令通过 `system` 角色注入，能有效引导大模型进入特定领域专家模式，相比 `user` 角色具有更高的指令优先级。
2. **指令详尽**：Prompt内容涵盖了角色定义、任务目标、输出格式要求，逻辑清晰，有助于获得结构化的评审结果。

#### 🤔问题点：
1. **硬编码大段文本**：将数百字的Prompt直接硬编码在Java类中，严重违反了关注点分离原则。这种写法导致代码臃肿，阅读困难，且每次修改Prompt都需要重新编译部署，缺乏灵活性。
2. **可维护性差**：字符串使用 `+` 进行拼接且包含换行符，极易在修改时引入格式错误，且难以直观看出Prompt的实际排版效果。
3. **资源管理缺失**：此类长文本Prompt应作为资源文件（如 `.md` 或 `.txt`）管理，或者存入配置中心/数据库，而非耦合在业务逻辑代码中。
4. **潜在的字面量错误**：虽然使用了 `Model.CHATGLM_5.getCode()`，但角色字段 "system" 仍为字符串字面量，建议封装或使用常量，防止拼写错误。

#### 🎯修改建议：
1. **Prompt 外部化**：立即将 Prompt 内容提取到外部资源文件中（如 `resources/prompts/code-review-system-prompt.md`），在 Service 层通过 IO 流或工具类读取。实现配置与代码解耦。
2. **使用 Java 文本块**：如果必须保留在代码中（不推荐），请使用 Java 15+ 引入的 Text Blocks (`"""`) 语法，以替代 `+` 号拼接和 `\n` 换行符，大幅提升可读性。
3. **异常处理**：读取外部资源文件时，务必添加异常处理机制，防止因文件缺失导致服务启动失败。
